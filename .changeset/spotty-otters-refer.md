feat: artifact references — `Artifact.add_reference(uri, name=None, checksum=True, max_objects=None)` records external data by URI in the artifact manifest *without* staging any blob, so large or already-durable datasets can be versioned, deduplicated, alias-promoted, and lineage-tracked without copying the bytes. A pluggable per-scheme handler resolves size + a checksum and expands a directory/prefix into one entry per object (capped by `max_objects`): `file://` (stdlib, sha256), `http(s)://` (bundled `httpx`), `hf://` (core `huggingface_hub`; LFS sha256 or git blob id), and `s3://` / `gs://` / Azure blob URLs (optional `trackio[s3]` / `trackio[gcs]` / `trackio[azure]`, lazily imported — degrading to an un-checksummed record when absent). Any other scheme is kept as an opaque pointer. `download()` fetches referenced objects (raising if unreachable, and warning — not failing — when a source's re-probed checksum no longer matches the recorded one); `references` / `get_entry_uri` expose the URIs without downloading. A URI whose query string looks like a signed URL (presigned S3/GCS, Azure SAS) triggers a warning before it is persisted, since manifests may sync to a shared HF Dataset. Each reference's checksum is stored in the single `digest` manifest field (wandb-style; opaque for references) — a sha256 for local/LFS references, the provider ETag for cloud/HTTP, or the URI itself when no checksum can be obtained (with a warning). Existing file-only manifests remain byte-identical, so their `manifest_digest` values are unchanged.
